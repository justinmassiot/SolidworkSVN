Solidworks Subversion Integration Requirements

Author: Carl Allendorph

Date: Feb 4 2012

Overview

The intent of this project is to integrate subversion revision control support into the Solidworks™ application as a replacement for the PDMworks or PDM vault solutions. This would allow the locking of parts to be automated and allow revision control to be implemented in a safe way when working with the solidworks binary files. This would allow multiple users to work on parts that are utilized in multiple assemblies to be shared and worked on simultaneously. This document is intended to outline the requirements for this integration effort and list the feature set for this software set. It will also describe the business model in terms of how the software would be sold to customers.

Audience

This document is intended for the engineering, QA, marketing, and business development departments. It is intended for anyone that needs to know the minimum feature set of the product.

Scope

This document is intended to describe the requirements for the software integration and the UI. It is not intended to make implementation decisions except where they influence the UI appearance and the intent of the design.

Background

PDMworks™ is a revision control vault system for Solidworks. It allows users to track changes to a mechanical design as it progresses. It also allows the user to control access to the components for read/write and controls changes that are made to released components. It implements a version of “Product Lifecycle Management” (PLM). This allows product development to be controlled from inception, to design, prototyping, qualification, and finally production. It allows files to be shared with many users simultaneously and reduces the amount of files that are shared using email or other uncontrolled methods. In the end, this leads to a more robust process that more safely stores data and prevents data loss.

Unfortunately, PDMworks has a variety of problems. First, it is a closed source implementation, so any bugs that do exist can’t really be fixed. Second, it does some very stupid things - like making the filename the key in the database that associates a part to its revision history. This means that parts can’t be renamed. Finally, it costs a fair amount of money. In general, small startups can’t afford the cost of this kind of solution even though the benefits of it are so superior to the alternatives.

Most smart startups instead look to using Subversion (SVN) or other revision control systems to meet their needs. With TortoiseSVN, this allows a fair bit of flexibility when implementing revision control even in the Windows environment. However, there are still challenges. Tagging of a given release version can be challenging at best. Because of the way that Solidworks searches for files, having any kind of hierarchical file structures becomes almost impossible.

This is where this software integration comes in. The intent is allow the user to more easily integrate their Solidworks development with a revision control system. Flexible tagging ability to make revision control easier. Transparent file locking will allow the user to focus on making the design better instead of causing conflicts at a later date and losing work.

Usage Pattern

The intended general use case would allow the user to look at all of the files in a particular repository and to check-out those files. “Check-out” means that a file is locked in such a way that only the locking user can modify the file. Once the user has successfully locked the file, the file would then be opened in. If the locked file is changed, then file’s indicator would change states indicating that the file has changes that have not been committed yet. When the user is satisfied, the user would then click on the assembly or file and select the “Check-in” option. This option would bring up a dialog box that would indicate the files that are associated with the assembly that is changing. It would show which files have been changed and will be committed. This dialog would have options that indicate whether the user wants to unlock the files after committing or if the files should remained locked. This would be the general workflow for a design engineer.

For the Release Manager, the workflow would be a little different. The release manager would need to transition sets of files from a design state to a released state. For the files in the trunk branch, the release manager would basically lock the files indefinitely as the ‘release manager’ user. Additionally, the revision number of each of the files associated with this file would need to be stored so that when a previously released version need to be checked out, the appropriate versions of a particular file can be found. Ideally, this revision information would be stored in the same file as the assembly file or part file. 

Example Features:

  • Check-out / Check-in  - Locking files to allow synchronous access.
  • Where Used - Find all assemblies a part is used in.
  • Associated Files - (Drawings, word documents, excel sheet, etc)
  • Automatic updating of parameters (revision, description, creator, creation date, etc)
  • Indicators on a file -  Locked, out of date (changes recorded on server), up to date,  etc
  • Indication of current user profile (designer, release manager, etc) - This is used to move files from current state to a release state and make tags.
  • Retrieve a particular revision of (Part, Assembly, drawing, etc)
  • Store address of multiple repositories.

Layout

This Solidwork Add-in’s layout will consist of two primary UI features. The first will be an additional task pane (area of the toolbox, design library, etc) and a new Solidworks application window menu (where File, edit, view, help, etc are found). In addition to these UI features, the add-in will add a new section in the “System Options” configuration window of Solidworks that will allow the user to change configuration settings for the SolidSVN add-in application.

Application Window Menu

The application window menu will be relatively simple for the first iteration. It will only have an option to allow the user to manage the repositories that are currently available. Selecting this option will cause a new dialog window to open.

As shown, this dialog will allow the user to add/edit/remove repositories. When a repository is added in this method, the repository becomes available for access through the task pane (see below). When the user selects the “add” or “edit” buttons, a configuration dialog is shown that allows the user to modify parameters for the repository:

When the user requests to remove a repository, this will only remove the configuration parameters for that repository from SolidSVN. It will not delete the local SVN repository that was created in the “working directory.” Additionally, any files that are locked will be inspected. If the file is unchanged from the latest revision in the SVN repository, then the lock will be released. If the locked file has pending changes, the file will maintain its lock. This lock will need to be released either via re-adding the repository and finishing this process, or by using TortoiseSVN to manually commit changes and release the lock.

Task Pane

The task pane section of the add-in will contain the majority of the application functionality. Through this interface, the user will have access to parts in the repository that is currently being used. The user will be able to inspect parts for their revision state as well as current status. Below shows a more detailed view of just the task pane. As shown in this view, the user may select which of the configured repositories he/she is currently using (“Repository #1” drop down menu). As a default policy, the add-in will not allow users to add entities of one repository to assemblies or drawings of another repository. This is to mitigate the ability for the user to create a repository that depends on another repository. This will be a configuration option that the user can disable if so desired. 

Below the Repository selection drop down menu, various buttons for general control of this repository will be available. This buttons will allow the user to implement a variety of tasks and utilities.

  1. Update - This tells the local repository to force an update to get the latest files from the SVN remote repository. Below is an example dialog that would launch when attempting to update.

  1. Find - Find any reference to a selected part in any of the assemblies or drawings contained in this repository. Useful for tracking down dependencies. Below is an example of the dialog window that would launch when a file is selected and the “Find” button is clicked:

  1. Commit All - Open the commit dialog to commit all changes to the remote repository. See Commit dialog section below.

Below the command buttons is the repository tree listing. In this tree listing, directories, sub-directories, parts, assemblies, and drawings would be viewable. Each of the separate types of objects would have it’s own icon. Overlay-ed on top of each icon would be a status indicator that shows the status of the file with respect to the subversion repository.

Example Status Indicators (not exhaustive):

  1. Not Controlled - Indicates the file is in the working directory on the local repository but has not been added to revision control yet. All other status indicators show that the file is under revision control.
  2. Added - Indicates that the file has been added to the local repository but has not been commited to the remote repository yet.
  3. Stable - Indicates that the file is not currently locked by anyone and it is the same revision as is on the remote repository
  4. User Locked Unchanged - Indicates that the file is currently locked by the user but has the same revision as is on the remote repository
  5. User Locked Changed - Indicates that the file is currently locked by the user but does have changes locally that have not been commited to the remote repository.
  6. Other Locked - This indicates that the file in question is currently locked by another user. This file can not be opened for write access. The other user may be another designer or could be the release admin.
  7. Conflict - This indicates that the file currently has a conflicting state with the file currently on the remote repository.

Though not shown in the image, the tree view of files in the repository would also likely implement the datagrid bindings so that tag information such as “Description”, “revision”, and “part number” could be shown with each of the parts as well.

The filter bar allows the user to search the repository for a given word or phrase. This filter bar will likely have configuration settings that allow the user to exercise more parametrized search options. When text is entered into the filter bar, the application will immediately begin using a binary search tree to limit the available parts still shown in the repository. When the “X” of the filter bar is clicked, the current search filtering will be disabled and the full repository shown.

When a file is selected (single left click), the “File Parameters” section will be populated with parameter information about the selected file. This will include information like description, part number, revision, and other pertinent information. It will also have information about who owns the lock on the file and its current status.

When the file is selected with a single right click, a option menu is launched as shown below:

For parts that are not already under revision control (“Not Controlled” state), this menu will consist of only a few items the most important of which is “Add to repository.” This instructs the application to add the selected file to the repository (“svn add”). It will then be ready to commit.

For parts that are already under revision control, this menu will have other options:

As shown, this menu will allow the user to perform various actions on the file.

Example Actions (not exhaustive):

  1. Open - open the part in the Solidworks application. If the user has the lock on the file, the file will be opened read/write. If the lock has not be acquired, the file will be opened read-only. Optionally, the user can configure the add-in to automatically retrieve a lock for any file that is opened in this manner. Additionally, the user could elect to drag and drop the part from the repository window to an empty Solidworks UI frame or over an existing part. Note however, that drag and drop to an assembly will cause the part to be inserted into the assembly.
  2. Check-out - This causes the add-in to attempt to lock the file for this user.
  3. Commit - This causes the application to launch the commit dialog window. This dialog window will allow the user to push local changes to the remote SVN repository.
  4. Add to Assy - This allows the user to insert this part into an open assembly. Additionally, the user could opt to drag and drop the part into an assembly.
  5. Get By Revision - This instructs the add-in to access a particular revision of the Solidworks assy, part, or file and download it from the remote repository.

When the user selects the commit action on a given assembly, part, or file or alternatively, selects the “Commit All” button, a new dialog will be launched that will allow the user to manage the commit of files to the remote repository:

This dialog shows the files that are currently being considered for committing to the remote repository. This dialog will allow the user to release all locks on files being commited or alternatively to select and choose which files are released and which locks are kept. The data grid shown would be tree style data grid for assemblies showing which files have changes that will be commited with the assembly and which files are unchanged but still part of the assembly. Various other parameter information can be configured to be shown as well.

When the user wishes to commit changes, the user selects the commit button. The progress bar will then be incremented as the process proceeds. If the process encounters an error or problem while attempting to contact the remote repository, the dialog will launch error messages as shown below:

As with normal SVN activity, this operation is atomic. In other words, if the commit operation fails in the middle of the operation, then all operations prior to the error that did succeed will not be applied and user will be left in the same situation as before attempting to commit.

Backend Operation

In addition to the UI parameters that will need to be added, there are various back-end operations that need to be handled. Specifically, when files are opened through the regular Solidworks “File->Open” method or through the “Edit Part in Assy” mode, these functions will need to be caught to make sure that files are not edited without the lock for a particular file being acquired. This will require callback registration with the Solidworks application.

For requesting a particular revision of an assembly or part, the application will need to close the currently open part, attempt the svn access, and then reopen the part/assy

Application will continually check for updates in the background to see if the local repository is out of date. This will change the little indicators to show that the file is out of date and needs to be updated.

When files are changed, the icon status needs to change to show that file has changes that need to be commited. Alternatively, if it is up to date, it needs to have an icon indicating that it is the latest version. 

Planning

Version 0.1

This version would attempt to first try to implement the layout of the UI so that it can show the files that currently available in the repository. It would primarily be a UI framework without the true subversion functionality.

Version 0.2

Add the subversion bindings. Allow for checkout, update, commit, and such. Setup subversion server and document process.

Version 0.3

Implement the locking functionality such that files can be locked for synchronous access.

Implement the parameters tree for each file so that we can show the revision and holder of lock and such.

Version 0.4

Bug Fixes and cleanup version. Create installer for the add-in. Release to select people for comments and testing. Beta Release.

Version 0.5

Add the user roles (designer vs. release manager). Need to have some examples of how to implement this on the server and documentation of how this setup is done.

Version 0.6

Implement the “Find in Assemblies” feature where all references to a particular part can be found.

Version 0.7

Checkout a previous revision of an assembly. Handle the fixing of broken references due to the subversion

Version 0.8

Bug fixes and cleanup. Beta Release #2

Version 0.9

Polish. Implement website for marketing and such.
